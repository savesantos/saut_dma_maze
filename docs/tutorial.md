# Development tutorial

How to add code to this workspace, package by package, with the official ROS 2
conventions. Read [README.md](../README.md#getting-started) first for the
one-time setup; this document picks up after `colcon build` succeeds on an
empty workspace.

---

## 0. Mental model

This repo is a [ROS 2 colcon workspace](https://docs.ros.org/en/humble/Tutorials/Beginner-Client-Libraries/Creating-A-Workspace/Creating-A-Workspace.html):

```
saut_dma_maze/
‚îú‚îÄ‚îÄ src/
‚îÇ   ‚îú‚îÄ‚îÄ maze_mdp/      # ament_python ‚Äî algorithms + ROS nodes
‚îÇ   ‚îú‚îÄ‚îÄ maze_bringup/  # ament_python ‚Äî launch files & config
‚îÇ   ‚îî‚îÄ‚îÄ maze_msgs/     # ament_cmake  ‚Äî custom .msg / .srv
‚îú‚îÄ‚îÄ build/  install/  log/   # generated by colcon, git-ignored
```

Two terminal commands you will type all day:

```bash
source /opt/ros/humble/setup.bash   # underlay (ROS 2 itself)
source install/setup.bash           # overlay  (this workspace)
```

The build tool is `colcon`. The Python packages use `ament_python`
(setuptools under the hood); the messages package uses `ament_cmake` +
`rosidl`.

## 1. `maze_mdp` ‚Äî algorithms + ROS nodes

This is where most of the work lives. It is a regular Python package that
`colcon` knows how to install.

### 1.1 Layout to grow into

```
src/maze_mdp/
‚îú‚îÄ‚îÄ maze_mdp/                 # the importable Python module
‚îÇ   ‚îú‚îÄ‚îÄ __init__.py
‚îÇ   ‚îú‚îÄ‚îÄ mdp.py                # MDP class: states, actions, P, R, gamma
‚îÇ   ‚îú‚îÄ‚îÄ value_iteration.py    # value_iteration(mdp) -> (V, pi)
‚îÇ   ‚îú‚îÄ‚îÄ sarsa.py              # sarsa(env, episodes, alpha, eps, gamma)
‚îÇ   ‚îú‚îÄ‚îÄ qlearning.py          # q_learning(env, ...)
‚îÇ   ‚îú‚îÄ‚îÄ simulator.py          # GridMaze micro-simulator (no ROS)
‚îÇ   ‚îî‚îÄ‚îÄ nodes/
‚îÇ       ‚îú‚îÄ‚îÄ __init__.py
‚îÇ       ‚îî‚îÄ‚îÄ policy_runner.py  # rclpy.node.Node that drives the robot
‚îú‚îÄ‚îÄ test/                     # pytest tests + linters
‚îî‚îÄ‚îÄ setup.py
```

### 1.2 Pure-Python first (no ROS)

Write `mdp.py`, `simulator.py`, `value_iteration.py`, `sarsa.py`,
`qlearning.py` and run them with plain `python3` from the workspace root.
This is the **micro-simulator validation** step (workplan #3) and
deliberately has zero ROS coupling ‚Äî keep it that way so the algorithms can
be unit-tested and reused.

### 1.3 Adding a ROS node

A ROS 2 node is a class extending `rclpy.node.Node`. Minimal skeleton in
`src/maze_mdp/maze_mdp/nodes/policy_runner.py`:

```python
import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist

class PolicyRunner(Node):
    def __init__(self):
        super().__init__('policy_runner')
        self.cmd_pub = self.create_publisher(Twist, 'cmd_vel', 10)
        self.timer = self.create_timer(0.1, self.tick)

    def tick(self):
        # consult policy, publish Twist
        ...

def main():
    rclpy.init()
    rclpy.spin(PolicyRunner())
    rclpy.shutdown()
```

Register it as an executable in `src/maze_mdp/setup.py` under
`entry_points`:

```python
entry_points={
    'console_scripts': [
        'policy_runner = maze_mdp.nodes.policy_runner:main',
    ],
},
```

After `colcon build --symlink-install && source install/setup.bash`:

```bash
ros2 run maze_mdp policy_runner
```

> **Heads-up** ‚Äî every new entry point requires a rebuild
> (`colcon build --packages-select maze_mdp`). Pure source edits to existing
> `.py` files do **not**, thanks to `--symlink-install`.

## 2. `maze_msgs` ‚Äî custom interfaces

Only touch this when you need a message/service that does **not** exist in
`std_msgs`, `geometry_msgs`, `nav_msgs`, etc. Typical candidates for this
project: `MazeGrid.msg`, `Policy.msg`, `SetGoal.srv`.

Reference: [Creating custom msg and srv files](https://docs.ros.org/en/humble/Tutorials/Beginner-Client-Libraries/Custom-ROS2-Interfaces.html).

Steps:

1. Create `src/maze_msgs/msg/MazeGrid.msg`:
   ```
   uint32 rows
   uint32 cols
   int8[] cells   # 0 = free, 1 = wall, 2 = goal
   ```
2. In `src/maze_msgs/CMakeLists.txt`, add:
   ```cmake
   find_package(rosidl_default_generators REQUIRED)
   rosidl_generate_interfaces(${PROJECT_NAME}
     "msg/MazeGrid.msg"
   )
   ament_export_dependencies(rosidl_default_runtime)
   ```
3. In `src/maze_msgs/package.xml`, add:
   ```xml
   <buildtool_depend>rosidl_default_generators</buildtool_depend>
   <exec_depend>rosidl_default_runtime</exec_depend>
   <member_of_group>rosidl_interface_packages</member_of_group>
   ```
4. In any consumer (e.g. `maze_mdp/package.xml`), add
   `<depend>maze_msgs</depend>` and import with
   `from maze_msgs.msg import MazeGrid`.

Rebuild the whole workspace after adding messages:
```bash
colcon build --symlink-install
source install/setup.bash
```

## 3. `maze_bringup` ‚Äî launch & config

A bringup package contains **no logic**: it only orchestrates nodes
defined in other packages.

### 3.1 Layout

```
src/maze_bringup/
‚îú‚îÄ‚îÄ launch/
‚îÇ   ‚îî‚îÄ‚îÄ alphabot_maze.launch.py
‚îú‚îÄ‚îÄ config/
‚îÇ   ‚îî‚îÄ‚îÄ params.yaml
‚îî‚îÄ‚îÄ setup.py
```

### 3.2 Install launch/ and config/ files

By default `ament_python` only installs the Python module. Tell it to also
install the launch files and YAML configs by editing
`src/maze_bringup/setup.py`:

```python
import os
from glob import glob
from setuptools import setup

package_name = 'maze_bringup'

setup(
    # ...
    data_files=[
        ('share/ament_index/resource_index/packages',
            ['resource/' + package_name]),
        ('share/' + package_name, ['package.xml']),
        (os.path.join('share', package_name, 'launch'),
            glob('launch/*.launch.py')),
        (os.path.join('share', package_name, 'config'),
            glob('config/*.yaml')),
    ],
    # ...
)
```

### 3.3 Minimal launch file

`src/maze_bringup/launch/alphabot_maze.launch.py`:

```python
from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
    return LaunchDescription([
        Node(
            package='maze_mdp',
            executable='policy_runner',
            name='policy',
            output='screen',
        ),
    ])
```

Run with:
```bash
ros2 launch maze_bringup alphabot_maze.launch.py
```

## 4. The inner development loop

```bash
# once per shell
source /opt/ros/humble/setup.bash
source install/setup.bash

# edit src/maze_mdp/maze_mdp/*.py ‚Äî no rebuild required
ros2 run maze_mdp policy_runner

# rebuild only when you touch:
#   - setup.py / entry_points
#   - package.xml
#   - CMakeLists.txt
#   - any .msg / .srv
colcon build --symlink-install --packages-select maze_mdp
source install/setup.bash
```

`--symlink-install` is what lets you edit Python sources without
rebuilding. Use it always.

## 5. Testing & linting

`ros2 pkg create` already added `test_copyright`, `test_flake8` and
`test_pep257` under `src/<pkg>/test/`. Add your own pytest-style tests as
`test/test_*.py` and run:

```bash
colcon test --packages-select maze_mdp
colcon test-result --verbose
```

For pure-Python iteration on the algorithms, you can also run pytest
directly without colcon:

```bash
cd src/maze_mdp
python3 -m pytest
```

## 6. Recommended starting order

1. **`maze_mdp/mdp.py`** ‚Äî `MDP` dataclass + `GridMaze` env in `simulator.py`.
2. **`value_iteration.py`** ‚Äî easiest to validate (deterministic answer).
3. **`sarsa.py` + `qlearning.py`** ‚Äî reuse the same env, plot learning curves.
4. Add unit tests under `src/maze_mdp/test/`.
5. Only **after** the algorithms work offline:
   - write the ROS node (`nodes/policy_runner.py`),
   - declare custom messages in `maze_msgs` if needed,
   - wire everything in a `maze_bringup` launch file,
   - run on the AlphaBot2.
